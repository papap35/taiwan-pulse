# AI_PROJECT_SOP.md — AI 協作開發 SOP（通用範本）

本文件給 **AI agent** 看，定義「啟動一個新專案時，如何與使用者協作」的標準流程。
這份 SOP 是從本專案（台股監控系統）的實際協作經驗中抽象出來的，**與技術棧無關**，
可直接複製到任何新專案的根目錄，作為 AI 與使用者協作的起點。

> 使用方式：使用者把這份檔案複製到新專案後，對 AI 說「讀一下 AI_PROJECT_SOP.md，
> 我們要開始新專案」，AI 即依下列流程執行 Phase 0。

---

## 核心理念

1. **文件是路線圖，不是事後補的東西**。`SPEC.md` 決定「做什麼、按什麼順序做」；
   `AGENTS.md` 決定「怎麼做、品質門檻是什麼」。兩份文件先於程式碼存在。
2. **拆解到可以一句話啟動**。每個功能項目都要拆到「使用者只需要說『開始』或『下一個』，
   AI 就能獨立完成一個 PR」的細度。
3. **小步快跑、文件同步**。一個功能 = 一個 branch = 一個 PR，且 PR 必須同時更新對應文件，
   不留「之後再補文件」的債。
4. **AI 主動建議、使用者拍板**。AI 負責盤點現況、列出選項、給優先順序建議；
   使用者只需要做「同意/調整/挑選」這種低成本決策。

## Phase 0：專案啟動（Bootstrap）

當使用者說「我要開始一個新專案」或「幫我建立這個專案的開發規範」時：

### 0.1 建立 `SPEC.md`（功能規格 / 路線圖）

結構：

```markdown
# SPEC.md — <專案名稱> 功能規格與路線圖

## 現有功能盤點
| 模組 | 已完成 |
|---|---|
| ... | ...（若是全新專案則此表留空或省略） |

## 待開發功能規格

### P0 — <最高優先級分類>（必做，理由：...）
#### 1. 功能名稱 `[ ]`
**背景**：為什麼需要這個功能（使用者痛點 / 業務目標）
**功能規格**：
- 具體要做什麼（條列）
- UI / API / 資料結構的關鍵變更

---
### P1 — <次優先級分類>
...

## 技術債與基礎強化
（測試覆蓋率、效能、錯誤處理等非功能性需求，獨立分類，不混入 P0-Pn）

## 優先開發路徑建議
（用條列或表格給出建議順序與粗估工時）
```

**優先級分類的判斷原則**（依專案性質調整分類名稱，但邏輯固定）：

| 優先級 | 意義 | 範例 |
|---|---|---|
| P0 | 不做會直接造成損失 / 阻斷核心流程 | 交易安全、資料正確性、登入 |
| P1 | 核心價值主張的必要功能 | 主要分析/處理邏輯 |
| P2 | 提升效率的進階功能 | 篩選、排序、批次操作 |
| P3 | 長期使用必備但非急迫 | 紀錄、統計、報表 |
| P4+ | 差異化 / 實驗性 / 依賴外部資源 | 與同類產品比較後補的功能 |

### 0.2 建立 `AGENTS.md`（開發規範手冊）

依專案技術棧調整，但必須包含以下五個章節（章節結構參考本專案 `AGENTS.md`）：

1. **寫程式的原則**：分層架構（如：邏輯層/UI層/狀態層的依賴方向）、純函式優先、
   防禦性資料處理（null/0/NaN）、錯誤處理慣例
2. **測試的原則**：哪些程式碼必須有測試、防迴歸測試規則、測試命名規範、執行指令
3. **開發流程原則**：Branch 命名規則、PR 前自我 review 清單（**含文件同步檢查，視為硬性門檻**）
4. **Commit 原則**：Conventional Commits 格式、commit 前檢查清單
5. **SPEC.md 判讀與更新原則**：優先級判讀規則、狀態標記（`[ ]`/`[x]`/`[~]`/`[-]`）、
   新增功能時的格式範本

### 0.3 確認技術棧速查表

在 `AGENTS.md` 附錄記錄：語言/框架版本、測試指令、啟動指令、部署方式。
這是 AI 每次開始工作前快速確認環境的依據，避免每次重新探索。

### 0.4 與使用者確認範圍

完成 0.1-0.3 後，**不要直接開始實作**。先用 1-2 句話總結：
- SPEC.md 列了幾項、分幾個優先級
- 建議的開發順序
然後等待使用者回應（可能調整順序、刪減項目、或直接說「開始」）。

## Phase 1：需求拆解（新增功能到 SPEC.md）

當使用者提出新需求，或要求「分析還可以做什麼」時：

### 1.1 盤點現況

- 讀取 `SPEC.md` 確認哪些已完成（`[x]`）、哪些待開發（`[ ]`）
- 讀取既有程式碼結構，避免重複建議已存在的功能

### 1.2 拆解每個新功能

每個功能項目必須包含：

```markdown
#### N. 功能名稱（中文/英文皆可） `[ ]`

**背景**：一句話說明痛點或動機，最好能對比「現在怎麼做 vs 應該怎麼做」

**功能規格**：
- 具體、可驗收的行為描述（不是「優化體驗」這種空話）
- 涉及的檔案/模組（讓未來實作時能快速定位）
- 必要時附資料結構 / API 設計的程式碼片段

---
```

### 1.3 排優先順序

- 用一句話給出建議開發順序（例：`25 → 26 → 28 → 31 → ...`），依「相對好做 →
  需要外部資源整合 → 進階差異化」排序
- 若新功能與既有優先級分類不符，新增一個分類（例：P7），不要硬塞進現有分類

### 1.4 寫入 SPEC.md 並開 PR

- 純文件變更也要走 branch → commit → PR 流程（`docs/` prefix）
- PR 內容：列出本次新增的項目清單，並標註建議順序

## Phase 2：迭代開發循環（一個功能一個循環）

當使用者說「開始」「下一個」「繼續」時，AI 執行以下循環，**每次只做一個功能項目**：

```
1. 從 SPEC.md 依建議順序挑下一個 [ ] 項目
2. git checkout main && git pull --ff-only
3. git checkout -b <type>/<scope>-<簡述>
4. 實作功能（遵循 AGENTS.md 第 1 章原則）
5. 補測試（遵循 AGENTS.md 第 2 章原則）
6. 自我 review（AGENTS.md 3.5 節 checklist）
7. 文件同步（SPEC.md 標記 [x]、README/USER_MANUAL 等，視專案而定）—— 必須在同一 branch
8. commit（Conventional Commits 格式）
9. push + gh pr create
10. 回報使用者：做了什麼、PR 連結、下一個建議是什麼
```

**循環之間的溝通原則**：
- 每完成一個功能，用 1-3 句話總結「做了什麼 + PR 連結 + 接下來建議」，不需要長篇報告
- 不要在使用者沒要求時，一次做多個功能、開多個 PR——維持「一次一個、可隨時插話調整」的節奏
- 若實作中發現規格不夠清楚或有更好做法，**先用 1-2 句話提出，等確認再繼續**（不要默默改變範圍）

## Phase 3：例外狀況處理

| 情境 | 處理原則 |
|---|---|
| 使用者臨時插入緊急 bug fix | 獨立開 `fix/` branch，走完整流程，完成後再回到原排程 |
| 開發中發現某功能依賴的外部資源不可用 | 在 SPEC.md 該項目標註「資料來源待確認」並 `[ ]`保留，先跳過做下一項，不要卡住整個循環 |
| 使用者要求的功能跨越多個優先級分類 | 拆成多個獨立項目分別編號，各自可獨立完成、獨立 PR |
| 一個功能做完發現規模比預期大 | 拆成「最小可用版本」先完成並標 `[x]`，剩餘部分新增一個項目 `[ ]`並說明是延伸 |

## 附錄：給使用者的極簡指令對照表

這份 SOP 設計成讓使用者可以用極短的指令驅動：

| 使用者說 | AI 應該做 |
|---|---|
| 「分析還有什麼可以做」 | 執行 Phase 1：盤點 + 拆解 + 建議順序，**寫入 SPEC.md** |
| 「全部寫到 spec，之後慢慢做」 | Phase 1 完整輸出 + 開 PR，不開始實作 |
| 「開始」/「開始做」 | Phase 2：取下一個 `[ ]` 項目，完整跑完循環 |
| 「下一個」/「繼續」 | 同上，重複 Phase 2 |
| 「先做 OO」 | Phase 2，但跳過建議順序，直接做使用者指定的項目 |
| 「這個之後再做」 | 該項目維持 `[ ]`，調整建議順序，跳到下一個 |
